# 0012 — Refresh v2 PIT archive when signal logic changes; v1 archive is frozen reference

**Status:** Accepted
**Date:** 2026-05-10
**Decided by:** Amit (with Claude Code)

## Context

v2 keeps two PIT snapshot archives:
- `daily_snapshots_pit_v1` — read-only copy of v1's archived snapshots, ~36 monthly periods. Frozen reference, never edited.
- `daily_snapshots_pit` — v2's own archive, currently 7 monthly periods, regenerated by `tools/reconstruct_pit.py`.

Today (2026-05-10) we found that `SCREEN["financial_sectors"]` had been set to `"Financial Services"` while `stocks.sector` actually contains `"Financials"` — the financial-sector exclusion was a silent no-op across 5 modules since v2 began. Fixed in commit `af94835`.

That fix changes signal logic going forward. The question: should we refresh the existing v2 PIT archive (`daily_snapshots_pit`) so it reflects the corrected logic, or leave the past as-is?

The reflexive answer earlier in the day was "leave it; archives are immutable" — based on the argument that v2's match to v1's t-stats validates the port. That argument was wrong, and the user pushed back: matching a buggy reference doesn't make the numbers correct, just consistently wrong.

## Decision

**Refresh v2's own PIT archive whenever signal logic changes in a way that would meaningfully alter the snapshot. The v1 archive stays frozen.**

Two distinct things were getting conflated:
- **Port correctness** (does v2 reproduce v1's logic faithfully?) — measured by `v2_recompute` t-stats matching `v1_archive` t-stats when run on equivalent data. This is a one-time check, useful at the rebuild boundary, then irrelevant.
- **Financial truth** (do the numbers measure what we want them to measure?) — measured by re-running v2's pipeline with corrected logic and looking at the resulting t-stats. This is what we actually care about.

After the rebuild boundary, financial truth is the only criterion. v2 should drift away from v1 whenever v2 is genuinely better than v1.

Operational rule:
1. Any signal-side fix that changes which stocks get scored, or how, triggers a `tools/reconstruct_pit.py` re-run.
2. After re-run, `tools/backtest_pit.py` re-computes `v2_recompute` t-stats with the corrected logic.
3. The `v1_archive` rows in `pit_ic_by_tier_v2` stay as-is — they're the historical reference, not the truth.
4. Promotion of new factors to scoring weights uses post-fix `v2_recompute` t-stats only.

## Why we got it wrong the first time

The "validation matches v1 exactly" line in earlier handoffs and CHANGELOG entries gave a false sense of correctness. It validated the port; it did not validate the signals. The correct framing in handoffs and changelogs going forward:

> "v1 archive matches v2 recompute when given equivalent inputs — this is a port-correctness check, not a financial-truth check. v2 numbers should diverge from v1 whenever v2 fixes a bug."

## What we ran today to honor this decision

Re-ran `tools.reconstruct_pit` (215 sec wall time, 7 dates × 26 signals × 2,448 stocks → 17,136 rows in `daily_snapshots_pit`). Re-ran `tools.backtest_pit` (132 rows in `pit_ic_by_tier_v2`).

Notable post-fix shifts vs pre-fix `v2_recompute`:
- `cf_accruals_ratio` MID: t=−2.89 → −1.93 (KEEP → WEAK band). Financials had been amplifying the signal; cleaner sample is honestly weaker.
- `bs_accruals_ratio` SMALL: t=−2.49 → −2.25 (small weaken).
- `piotroski_f_score`, `z_score`, most other quality signals barely moved.

This is the actual, post-fix signal quality. Future weight assignments use these numbers.

## Alternatives considered

- **"Archives are immutable, fix forward only."** Rejected. Sounds operationally tidy but produces wrong t-stats forever. The archive is cheap to regenerate (≤4 minutes); refusing to do so is performative caution.
- **"Refresh v1 archive too."** Rejected. v1's archive is v1's output — we don't own v1's pipeline. Keeping it frozen as a reference point preserves a useful historical anchor.
- **"Refresh only when the bug is severe."** Rejected — too subjective. The right rule is mechanical: signal logic changes → archive refresh. Cost is low, benefit is correct numbers.

## Consequences

- After every signal-side fix, the routine is: fix → reconstruct → backtest → review t-stat deltas. Add to `tools.reconstruct_pit` as part of the standard fix-up checklist.
- `v2_recompute` t-stats in `pit_ic_by_tier_v2` are always "the current truth as of the last reconstruction run." `v1_archive` rows are the historical reference.
- The "matches v1 exactly" framing should be retired from go-forward documentation. It was useful at the rebuild boundary and is now misleading.
