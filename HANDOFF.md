## Don't do

- **Don't fuzzy-match unlisted-player names back to listed tickers.** The exact bug fixed in `903de71`: name-token fallback mapped "Reliance Jio" → `RCOM`. Only explicit-ticker enrichment is safe. If `competitive_landscape.players[i].ticker` is null in the narrative, leave the player as non-clickable Private. **Why:** Group-name prefixes (Reliance, Tata, Bajaj, Adani) shadow multiple unrelated listed entities; a wrong click sends users to the wrong stock and is worse than no click.
- **Don't pkill -f "uvicorn cockpit.app".** Pattern matches the systemd-managed prod cockpit on port 3000 too. Use `sudo systemctl restart alpha-cockpit.service` for prod, or kill test instances by PID / by port. Memory note: `feedback_dont_pkill_cockpit.md`.
- **Don't rename `sector_metadata.sector` column without an ADR.** Legacy rows (sector-keyed) and new rows (industry-keyed) both use it as a generic taxonomy key. PK `(sector, source)` is referenced from `cockpit/api.py:get_sector_metadata` and `get_industry_metadata` (which delegates). Rename touches the schema, every cockpit read path, and the fetcher upsert. Pure tech debt — not blocking anything.
- **Don't re-run the narrative fetcher with `--skip-existing` removed.** It would re-charge ~₹30 to overwrite well-formed narratives with stochastic re-generations. The `sector_metadata` rows from today are good. If a single industry needs refresh, use `--industry "X"` to target just that one.
- **Don't commit the screenshots in repo root.** Move them to `docs/_archive/playwright-screens/` or add `*.png` to `.gitignore` (with allowlist exception for any genuinely needed in-repo images). Listed under "half-built" — clean up before next commit batch.
- **Don't run `screener_pull` and `screener_schedules` concurrently.** Same cookie, doubled request rate, easy way to trip Screener's bot detection. Carried-forward rule.
- **Don't add a name→ticker mapping for subsidiary→parent (e.g. Jio → RELIANCE) without a deliberate decision.** The clean fix is the narrative providing a `parent_ticker` field; the messy fix is a hard-coded lookup. Either way it's an ADR-worthy decision, not a passing edit.
- **Don't ship a factor module without a PIT helper alongside.** Carried-forward rule from 2026-05-10. The pair is the unit of work.
- **Don't edit `SCREEN.weight_tiers` mechanically when shipping a factor.** Carried-forward rule.
- **Don't reintroduce `tools/apply_splits.py` or `stock_prices.adj_close`.** Per ADR 0010.
- **Don't switch `fundamentals_screener` to wide format.** Per ADR 0011.
- **Don't rename `stocks.market_cap_cr` globally without an ADR.** Carried-forward.
- **Don't `git commit --amend` / `git add .` / `git add -A`.** Carried-forward.

## Open questions for me (decisions you need to make)

1. **File an ADR on "drill by industry, not GICS sector"?** Today's design decision affects schema (`stocks.industry` is now load-bearing), cockpit routes (`/sectors?industry=` is the primary, `?sector=` is back-compat), and downstream factor work (plan 0007's "market share momentum" presumes industry granularity). ~15 min ADR. **My take:** yes, ADR 0013. The decision is the kind of thing future-me would want a recorded rationale for, especially the choice to overwrite GICS sector tags from the from-scratch classifier (Eternal/Zomato fix).
2. **File an ADR on "competitive_landscape includes private players, sourced from LLM"?** Three things worth recording in one ADR: (a) including private players is a deliberate departure from "we only model the listed universe" — for *understanding* the industry it's essential, for *picking stocks* it's noise; (b) the data source is an LLM with web search, not a maintained dataset — so the values are best-effort, dated by `as_of`, and refresh cadence matters; (c) why we trust the values enough to put them in the UI without disclaimers. ~10 min ADR. **My take:** yes, ADR 0014.
3. **Market-cap-weighted composite vs equal-weighted — record the choice?** Today's switch makes the composite reflect index weight, which is what an investor sees in their portfolio. But equal-weighted has the property of surfacing breadth (whether the small fries are also doing well, not just the giants). Two valid choices. **My take:** ADR-worthy if I plan to keep mcap-weighted as the default. ~5 min ADR. Or fold it into ADR 0013 as a sub-section.
4. **Cron the narrative fetcher monthly?** Industry narratives reference 2024-2025 figures today. Run quarterly to keep them fresh? Or trigger manually when something material changes (a regulator move, a major IPO that redraws competitive_landscape)? **My take:** quarterly cron is fine — cheap, low-risk, fetcher is idempotent on `(sector, source)`. Monthly is wasteful. Quarterly Sundays at 02:00 IST, separate from the data pipeline.
5. **Plan 0007 (Market Share Momentum) timing — start once we have a second snapshot, or wait for full F-track batch first?** Plan 0007 needs two narrative snapshots ≥1 month apart to compute share-shift factors. F-track Phase B is the higher-priority workstream right now. **My take:** F-track first — share-shift is a niche signal, not a foundational one.
